Unit I

OOSE Fundamentals & SDLC Models

CO1

Core Definitions

  • Software Engineering: Disciplined, engineering-principles-based approach ensuring reliability, efficiency & maintainability of software.
  • OOSE: Develops software using OO principles — models system as interacting objects (data + behavior). Ensures scalability, flexibility.
  • OO Paradigm: Organizes software around objects combining attributes & methods. Improves modularity, reusability, maintainability.
  • SE vs. Programming: SE = systematic process (analysis→design→test→maintenance); Programming = mainly coding.

OOP Benefits & Principles

  • Reusability — code reused via inheritance
  • Maintainability — changes isolated per class
  • Encapsulation — data hidden inside objects
  • Polymorphism — same interface, multiple behaviors
  • Abstraction — expose essentials, hide details
  • Scalability — add new classes without major changes

Waterfall Model

  • Type: Linear, sequential, top-down
  • Approach: Each phase must complete before next starts
  • Phases: Requirements → Design → Implementation → Testing → Deployment → Maintenance
  • Advantages: Simple to manage, heavy documentation, clear milestones
  • Limitations: Inflexible, changes are costly, errors detected late
  • Best for: Small/simple projects with stable, well-defined requirements

Spiral Model

  • Type: Iterative, risk-driven, prototype-based
  • Each cycle: Planning → Risk Analysis → Engineering → Evaluation
  • Risk focus: Risk analysis in every iteration — identifies & mitigates early
  • Advantages: Handles uncertainty, continuous feedback, risk reduction
  • Best for: Large, complex, high-risk projects (banking, aerospace)

Agile Methodology

  • Definition: Flexible, iterative development focused on collaboration, customer feedback, rapid delivery
  • Iteration (Sprint): 1–4 week cycles, each delivering a working increment
  • Role of feedback: Early error detection, continuous improvement, better alignment with requirements
  • Advantages over Waterfall: Flexible to change, faster delivery, customer-centric

Unit II

UML, Requirements Analysis & System Modelling

CO2

Requirements Types

  • Functional Requirements: What the system should do. Ex: "Allow fund transfer, view balance."
  • Non-Functional Requirements: How well it performs. Ex: "Support 10,000 concurrent users, 99.9% uptime."
  • Evaluation criteria — Functional: Clear, complete, feasible, verifiable
  • Evaluation criteria — Non-functional: Measurable, consistent, verifiable (performance, security, reliability)

UML — Three Ways to Apply

  • As a Sketch: Informal, quick diagrams to explore ideas
  • As a Blueprint: Formal detailed diagrams guiding construction
  • As a Programming Tool: Generate code from models using CASE tools

Use Case Diagram Elements

  • Actor: External user/system (stick figure)
  • Use Case: Function/service (oval/ellipse)
  • System Boundary: Rectangle showing scope
  • Association: Line connecting actor & use case
  • Include/Extend: Dashed arrows for mandatory/optional behavior
  • 3 Actor Types: Primary, Supporting, Offstage

UML Relationships

RelationshipSymbolMeaning
Association——Basic connection. Ex: Passenger ↔ Ride
Aggregation◇—Whole-part; parts can exist independently. Ex: Dept ◇— Employee
Composition◆—Strong whole-part; parts can't exist without whole. Ex: ATM ◆— Cash Dispenser
Inheritance→ (hollow)Subclass inherits superclass. Ex: Staff → Doctor
Generalization→ (hollow)Superclass specialized into subclasses

Node & DFD

  • Node (Deployment Diagram): A physical element representing a hardware device or execution environment where software components are deployed. Ex: ATM Machine node ↔ Bank Server node.
  • DFD Elements: Process (circle/rectangle), Data Store (open rectangle), Data Flow (arrow), External Entity (square). Shows how data moves through system.

Access Modifiers in Class Diagram

  • + (Public): Accessible from anywhere
  • - (Private): Accessible only within the class
  • # (Protected): Accessible within class and subclasses
  • Example: - balance: double (private attr), + getBalance(): double (public method), # accountId: int (protected attr)
  • Package: Group related classes, show dependencies. Ex: HMS — Patient Services, Billing, Laboratory
  • Component: Software components & interactions. Ex: Ride-Sharing — Ride Mgmt, Payment, Notification
  • Deployment: Physical hardware & software nodes. Ex: ATM machine ↔ Bank Server

Dynamic Modelling Techniques

  • Dynamic Modeling: Focuses on system behavior over time — interactions, message flow, state changes
  • Sequence Diagram: Shows interaction between objects in time order. Helps understand object collaboration, use case realization, detect missing interactions
  • State Transition Diagram: Shows states of an object & transitions due to events. Represents object life cycle, models event-driven behavior
  • DFD: Graphical representation of data movement — processes, stores, flows, external entities

Component & Deployment Diagram Analysis

  • Component Diagram — Dependencies: Shows provided/required interfaces, identifies tight coupling, detects dependency cycles, supports impact analysis
  • Component Diagram — Reusability: Identifies loosely coupled components with high cohesion and generic interfaces
  • Deployment Diagram — Performance: Identifies load per node, detects network latency, locates bottlenecks
  • Deployment Diagram — Scalability: Visualizes distributed nodes, supports horizontal/vertical scaling
  • Deployment Diagram — Security: Identifies critical nodes, analyzes network boundaries & firewalls

Package Diagram & Dependencies

  • Purpose: Organizes elements into packages, shows dependencies
  • Scalability: Loose coupling → easy expansion; cyclic dependencies → reduce scalability
  • Maintainability: Clear deps → easier debugging; tight coupling → more maintenance effort
  • Reusability: Independent packages → reusable; highly dependent → reduces reuse
  • Critical: Poor design leads to dependency explosion, rigid & error-prone system

Recommendation System (Case Study Key Points)

  • Stakeholders: End Users, Business Owners, Data Scientists, Developers, Sys Admins, Marketing
  • Functional Req: User registration, collect behavior data, generate recommendations, dynamic updates
  • Non-functional: Performance, scalability, security, 24/7 availability, accuracy
  • User Profiling: Registration → store details → maintain history → update dynamically
  • Rec Engine Steps: Data Ingestion → User Profiling → Candidate Generation → Scoring → Re-ranking → Feedback loop

OO Design, Patterns & Architectural Styles

CO3

Design Patterns — Overview

  • Pattern: Reusable solution to a recurring problem
  • Design Pattern: Proven solution for common software design problems
  • Uses: Improve reuse, flexibility, maintainability; provide standard solutions; reduce dev time
  • Categories: Creational, Structural, Behavioral

Behavioral Patterns (List)

  • Observer, Strategy, Command, State, Iterator, Mediator, Memento, Visitor, Template Method, Chain of Responsibility

Key Design Patterns

PatternTypePurpose & Use Case
SingletonCreationalOnly one instance; global access point. Use: DB connection, Logger
Factory MethodCreationalInterface for object creation; subclasses decide class. Use: Flexible object creation
ObserverBehavioralOne-to-many dependency; subject notifies all observers on state change. Use: Event-driven systems, GUI
AdapterStructuralBridge between incompatible interfaces; enables reuse of legacy code. Use: Integrating third-party libs
BridgeStructuralSeparates abstraction from implementation

Coupling & Cohesion

  • Coupling: Degree of dependency between modules. Low coupling = desirable (independent, easy to maintain)
  • Types of Coupling: Content, Common, Control, Stamp, Data
  • Cohesion: Degree to which module elements belong together. High cohesion = desirable (single responsibility)
  • Goal: Loosely coupled + highly cohesive → easy to modify, maintain, test, scale

GRASP Patterns

  • GRASP: General Responsibility Assignment Software Patterns
  • Creator: Assign creation to class with most knowledge
  • Information Expert: Assign responsibility to class with needed info
  • Low Coupling: Minimize class dependencies
  • High Cohesion: Focused, related responsibilities per class
  • Controller: Centralized class handles system events
  • Pure Fabrication: New class fulfills responsibility without violating cohesion
  • Indirection: Use intermediaries to decouple classes
  • Polymorphism: Multiple implementations via inheritance/interfaces

Adapter Pattern — Pros & Cons

  • Pros: Promotes code reuse; decouples system from implementation; supports multiple interfaces; isolates adaptation logic
  • Cons: Adds complexity; slight performance overhead; multiple adapters → maintenance effort; risk of overuse
  • Use when: Enabling incompatible systems to communicate; reusing existing code/libraries without rewriting
  • Avoid when: System is already compatible; performance-critical; project is too short-lived

Observer Pattern — Features

  • Loose Coupling: Subject doesn't know observer details
  • Dynamic Relationships: Observers added/removed at runtime
  • Scalability: Works with many dependent objects
  • Automatic Synchronization: State change propagates to all observers
  • Real-world: Stock market apps, GUI event systems, notification services

Unit IV

Implementation, Testing & Debugging

CO4

Core Definitions

  • Software Testing: Critical element of SQA; ultimate review of specification, design & coding. Finds errors, ensures reliability.
  • Verification: "Building the product right?" — ensures correct implementation of specific functions
  • Validation: "Building the right product?" — ensures software is traceable to customer requirements
  • Objectives: Execute program to find errors; good test = high probability of finding undiscovered errors

Testing Types Hierarchy

  • White Box Testing: Internal structure/code known to tester; verifies logic, paths, loops, conditions
  • Black Box Testing: Internal structure unknown; tests functionality (inputs → outputs) from user perspective
  • Gray Box Testing: Partial knowledge of internal structure
  • Functional: Tests against functional requirements → Unit, Integration, System Testing
  • Non-Functional: Quality attributes → Performance, Usability, Compatibility Testing

Testing Levels

LevelScopeKey Points
Unit TestingIndividual components in isolationUses stubs/drivers/mocks; fast, early detection; can't catch integration errors
Component TestingIndividual components independentlyTests usability, input/output behavior before integration
Integration TestingCombined modules as a groupAfter unit, before system testing. Approaches: Non-incremental, Incremental
System TestingComplete integrated systemRecovery, Security, Stress, Performance testing
Usability TestingEnd-user perspectiveNon-functional; tests user-friendliness, usefulness; performed during system & acceptance testing

Integration Testing — Top-Down vs Bottom-Up

  • Top-Down: Starts from the top module; uses stubs for lower modules; breadth-first or depth-first
  • Bottom-Up: Starts from the lowest modules; uses drivers for higher modules; no stubs needed
  • Non-incremental: All modules integrated at once ("big bang")

Debugging Approaches & Stages

  • Approaches: Brute Force, Backtracking, Cause Elimination, Program Slicing, Debugging Tools
  • Stages: 1) Identify error → 2) Locate error → 3) Analyze cause → 4) Design fix → 5) Correct error → 6) Recompile → 7) Test fix (regression) → 8) Document fix
  • Locating tools: Backtracking, program slicing, debuggers with breakpoints

Regression Testing

  • Definition: Confirms no new defects introduced after changes; old defects remain fixed
  • When to perform: After new feature addition, requirement change, defect fix, performance fix, environment change
  • Techniques: Re-test All; Regression Test Selection (reusable vs obsolete test cases); Prioritization (by business impact)

Test Case Management

  • Definition: Process where testing activities are managed to ensure high quality testing of a software application
  • Consists of: Tracking, organization, controlling process, checks visibility of the testing process
  • Goal: Deliver high quality software application
  • Key aspects: Organizing test cases, assigning to testers, tracking execution status, reporting defects, managing test cycles
  • Definition: Tools execute tests, report outcomes, compare with earlier results
  • Benefits: Higher accuracy, better reporting, increased coverage, improved bug detection, reusability
  • Best for automating: Repetitive tests, tests with multiple data sets, tests impossible to perform manually, tests prone to human error
  • vs Manual: Manual gives human insight per step; automation is faster for regression and repetitive tests

Usability Testing

  • Type: Non-functional, black-box; tests user-friendliness, usefulness, traceability, desirability
  • When: Early stages of SDLC; throughout system & acceptance testing
  • Advantages: Validates usability, improves satisfaction, discovers usability issues before delivery
  • Disadvantages: Costly (lab setup, engineers), harder to find volunteers, not 100% representative of real conditions

Testing Activities & Objectives

  • Testing Activities (5): Test Planning → Test Case Design → Test Execution → Data Collection → Effective Evaluation
  • Objectives of Testing: Execute program to find errors; A good test = high probability of finding undiscovered error; A successful test = uncovers an as-yet undiscovered error

Test Plan — 7 Steps (IEEE 829)

  • 1. Analyze product structure & architecture
  • 2. Design the test strategy
  • 3. Define all test objectives
  • 4. Define the testing area (scope)
  • 5. Define usable resources
  • 6. Schedule all activities appropriately
  • 7. Determine all test deliverables

Documentation Testing — Types

  • Test Scenario: Multiple ways/combinations to test; shows application flow; no inputs/navigation steps
  • Test Case: Step-by-step procedure with navigation, inputs, and expected output
  • Test Plan: All testing activities — objectives, scope, approach, environments, methodology, risks
  • RTM: Requirement Traceability Matrix — links requirements to test cases
  • Bug Report: Records detected defects

System Testing Types

  • Recovery Testing: System's ability to recover from failures
  • Security Testing: Verifies protection against improper penetration or data alteration
  • Stress Testing: Determines breakpoint / maximum service level
  • Performance Testing: Evaluates runtime performance, especially real-time software

Project Management, Agile, DevOps & Version Control

CO5

Software Project Planning

  • Definition: Identifying goals/scope, estimating cost/effort/time, allocating resources, identifying risks & mitigation plans
  • Risk vs Uncertainty: Risk = known unknown (predictable, can be planned); Uncertainty = unknown unknown (cannot be predicted/controlled)
  • Types of Risk: Technical, Schedule, Cost, Resource, External

Risk Management

  • Benefits: Reduces project failure; improves decision-making; enhances stakeholder confidence; optimizes cost & time
  • Mitigation Strategies:
  • Avoidance: Modify plan to eliminate risk. Ex: Use proven tech
  • Mitigation: Reduce probability/impact. Ex: Code reviews & testing
  • Transfer: Shift risk to third party. Ex: Outsource complex modules
  • Acceptance: Accept low-impact unavoidable risk. Ex: Minor perf issues

Version Control — Git & GitHub

  • Git: Distributed VCS — tracks changes locally; branching, merging, complete history
  • GitHub: Cloud platform hosting Git repositories; enables pull requests, issues, code reviews, collaboration
  • Git Features: Distributed VCS, branching & merging, tracks changes, collaboration support (via GitHub)
  • Git vs GitHub: Git = version control system (local); GitHub = cloud platform (remote collaboration)
  • Why important: Parallel development, conflict resolution, code history, team consistency

Git Workflow (Feature Branch Model)

  • 1. Clone remote repository
  • 2. Create feature branch from main
  • 3. Make changes & commit locally
  • 4. Push changes to remote (GitHub)
  • 5. Raise Pull Request for review
  • 6. Code reviewed → merged to main
  • 7. All team members pull updated code

Software Configuration Management (SCM)

  • Definition: Process of tracking & controlling changes in software to ensure consistency, integrity & traceability
  • Configuration Identification: Identify & uniquely name/version all Configuration Items (CIs)
  • Configuration Control: Manage changes via Change Request → Impact Analysis → Approval (CCB) → Implementation
  • Status Accounting: Record & report current status/history of CIs (e.g., Git logs)
  • Configuration Auditing: Verify software matches documented requirements; functional & physical audits
  • Role in Quality: Consistency, error reduction, traceability, supports QA audits, facilitates maintenance

Scrum Framework

  • 3 Pillars: Transparency, Inspection, Adaptation
  • Roles: Product Owner (manages backlog, prioritizes), Scrum Master (facilitates, removes obstacles), Development Team (design, develop, test)
  • Artifacts: Product Backlog (requirements list), Sprint Backlog (tasks for sprint), Increment (working software)
  • Sprint: 2–4 week cycle; each delivers a working increment
  • Events: Sprint Planning → Daily Standup → Sprint Review → Retrospective → Backlog Refinement

CI/CD Pipeline

  • CI (Continuous Integration): Auto-builds & tests code after every commit; detects integration issues early
  • CD (Continuous Delivery/Deployment): Auto-deploys validated builds to staging/production
  • Pipeline stages: Source Code Repo → CI (Build + Unit Tests) → Automated Testing → Artifact Repo → CD (Deploy) → Monitoring & Feedback
  • Impact on delivery speed: Eliminates manual bottlenecks, early bug detection, parallel development, continuous releases, faster feedback
  • Tools: Jenkins/GitHub Actions (CI), Docker (containerization), Prometheus/Grafana (monitoring)

DevOps Contribution to OOSE

  • Supports Modularity: CI/CD pipelines allow independent build/deployment of OO modules
  • Continuous Testing: Automated unit tests verify each class & method, preserve encapsulation
  • Version Control of OO Artifacts: Git manages class files, interfaces, UML diagrams, design documents
  • Faster Feedback: Monitoring tools help refactor classes, optimize object interactions
  • Enhanced Collaboration: Bridges designers (UML), developers (OOP), and operations (deployment)
